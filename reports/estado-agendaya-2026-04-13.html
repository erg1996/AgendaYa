3. Análisis FODA

Fortalezas

  • Código modular mantenible por 1 persona.
  • Multi-tenant bien modelado desde el día 1.
  • Hardening de seguridad al nivel de productos maduros.
  • Integración WhatsApp con plantilla personalizable y tokens firmados.
  • Cobertura de pruebas unitarias 31 / 31.
  • Contexto local del mercado salvadoreño por el fundador.

Oportunidades

  • PyMES sin digitalizar: barberías, salones, clínicas, spas, talleres.
  • Pagos locales (transferencia, Wompi) sin penetración de tarjetas.
  • Directorio público con SEO local como network effect.
  • Expansión a GT / HN / NI con el mismo código.
  • Sector gobierno / ONGs con agenda de atención ciudadana.

Debilidades

  • Founder único — bus factor 1.
  • SQLite limita escala horizontal (no multi-instancia con writes).
  • Sin procesador de pagos integrado.
  • Sin app móvil (web responsiva solamente).
  • Marca y SEO inexistentes; cero tráfico orgánico.
  • Registro público sin captcha / invite gate.

Amenazas

  • Competidores internacionales (Calendly, Fresha, Booksy) con presupuesto de marketing.
  • Copias rápidas por "vibe coders" locales.
  • Google Calendar + WhatsApp manual como "gratis" percibido.
  • Cambios de política en WhatsApp Cloud API (Meta).
  • Eventos macro — tipo de cambio, regulaciones de datos en SV.

4.2. Fixes críticos del scan de seguridad

#SeveridadCategoríaArregloArchivo
1críticoSeguridad CSRF bypass: se excluían todos los endpoints de /api/auth. Ahora solo login y register. CsrfMiddleware.cs
2críticoSeguridad Upload: whitelist MIME → extensión canónica, validación por magic bytes, assert de ruta absoluta dentro de uploadsDir. UploadController.cs
3críticoSeguridad User enumeration por timing en login: siempre se ejecuta BCrypt.Verify contra un hash dummy si el email no existe. AuthService.cs
4críticoPerformance Dashboard analytics cargaba la tabla completa en memoria. Ahora agrega en SQL (CountAsync, SumAsync, GroupBy). AnalyticsService.cs, AppointmentRepository.cs
5altoLógica Double-booking bajo concurrencia: check + insert ahora corren en una transacción serializable de SQLite. AppointmentRepository.cs
6altoLógica Recordatorios idempotentes: claim atómico con ExecuteUpdateAsync condicional. Si varias instancias corren en paralelo, solo una envía el email. ReminderBackgroundService.cs
7altoSeguridad JWT: ya no hay fallback silencioso a la dev key en producción. Lanza InvalidOperationException si falta Jwt__Secret. DependencyInjection.cs, Program.cs

4.3. Feature nuevo — Confirm / Cancel por link de WhatsApp

Valor al cliente: ataca directamente el dolor #1 real de barberías, salones y clínicas: los no-shows. El recordatorio por WhatsApp ahora incluye dos botones — confirmar o cancelar — que el cliente puede tocar sin loguearse, sin instalar nada. Los dueños ven el estado actualizado al instante en su panel.
  • Token HMAC-SHA256 firmado con clave derivada del Jwt:Secret (domain separation via HKDF-style).
  • Expiración corta: 48 horas tras generación del recordatorio. Un token filtrado queda obsoleto rápido.
  • Sin estado: no se guardan los tokens en BD. La validación es pura criptografía.
  • Páginas de confirmación HTML minimales servidas directo desde el controlador — sin dependencia del frontend.
  • Idempotente: clicar dos veces muestra el mismo estado final sin duplicar efectos.
  • Nuevos placeholders en plantilla WhatsApp: {confirmar_url} y {cancelar_url}.

Gaps de cobertura identificados

  • Lógica de solape ahora vive en el repo — los tests de unidad ya no pueden validarla (requiere BD real o SQLite in-memory). Recomiendo añadir tests de integración con Microsoft.EntityFrameworkCore.Sqlite en modo :memory:.

  • Sin pruebas E2E ni de controladores (WebApplicationFactory). Aceptable para MVP con 1 founder; priorizar después del primer cliente real.

8. Cómo desplegar (hoy mismo)

# 1. Generar un secreto JWT fuerte
export JWT_SECRET=$(openssl rand -base64 48)

# 2. Configurar el dominio del frontend y la URL pública del API
export FRONTEND_ORIGIN=https://tu-dominio.com
export PUBLIC_URL=https://api.tu-dominio.com

# 3. Levantar el stack
docker compose up -d --build

# 4. Verificar healthcheck
curl -f http://localhost:8080/health
Antes del lanzamiento público, aún falta: (a) proveedor SMTP real configurado (Brevo / Resend), (b) captcha o invite-only en registro, (c) certificado TLS via Cloudflare o Caddy delante del API, (d) probar el build Docker en la máquina local, (e) backup diario de scheduler.db a object storage.

9. Veredicto

La aplicación pasó de "MVP funcional con huecos de seguridad críticos" a "MVP hardeneado con feature de foso inicial y suite de pruebas completa" en esta sesión. El estado actual permite abrir una beta cerrada con 3-5 negocios en Sonsonate sin riesgo material. El siguiente movimiento de mayor retorno es el de reseñas post-cita — es la pieza que convierte a AgendaYa de "herramienta" a "activo imposible de abandonar".
